# 部署指南

## 生产前提

- Docker Engine 与 Docker Compose v2。
- 外部 Nginx 或 CDN 负责 HTTPS、域名和 SSE 反向代理。
- 仅 `/app/instance` 允许写入并挂载持久化卷；镜像其余文件系统只读。
- 生产环境不得启用 `APP_DEBUG`。

复制 `.env.example` 为 `.env`，至少配置：

- `SECRET_KEY`：独立生成、长度不少于 32 字符。
- `AI_API_KEY`：当前模型供应商密钥。
- `AI_GLOBAL_DAILY_LIMIT`：全站每日 AI 调用硬上限，必须为正整数。
- `AI_BASE_URL`、`AI_MODEL`：OpenAI 兼容端点和模型名。
- `TRUSTED_PROXY_HOPS`：应用前可信反向代理层数，单层 Nginx 使用 `1`。

密钥只通过部署环境注入，不得写入镜像、Nginx 配置或仓库。源码历史中发现的旧模型密钥必须在对应供应商控制台撤销。

## 构建与启动

镜像使用不可变版本号，不使用 `latest`：

```bash
export IMAGE_TAG=4.1.0
docker compose --env-file .env -f compose.prod.yml build
docker compose --env-file .env -f compose.prod.yml up -d
```

Compose 默认仅监听宿主机 `127.0.0.1:8000`。验证：

```bash
curl --fail http://127.0.0.1:8000/healthz
curl --fail http://127.0.0.1:8000/readyz
docker compose -f compose.prod.yml ps
```

`/healthz` 只表示进程存活；`/readyz` 同时验证生产配置与 SQLite 可用性，上线流量只能转发给就绪实例。

## Nginx 与 SSE

`flaskapp` 是外部 Nginx 反向代理示例。生产环境需另行配置 HTTPS 证书或置于 CDN 后方。`/api/lunming/stream` 必须保持：

- `proxy_buffering off` 与 `proxy_cache off`；
- HTTP/1.1；
- 不低于模型超时的 `proxy_read_timeout`；
- `X-Forwarded-For`、`X-Forwarded-Proto` 和 `X-Request-ID` 转发。

## 数据构建、备份与恢复

- `database/reference.db` 存储汉字笔画、383 条签文和彭祖百忌，是镜像内只读数据。
- `instance/runtime.db` 存储黄历缓存、联网笔画缓存和 AI 额度，是唯一需要持久化的数据文件。
- 编辑源保留为 Excel、JSON 等文件；发布前运行 `python scripts/build_reference_db.py` 重建基础库。
- 支持黄历日期范围为 `1900-01-01` 至 `2100-12-31`，旧黄历缓存不迁移。

备份时使用 SQLite 在线备份或先停止容器后复制卷，不得在写入期间直接复制数据库文件。停机备份示例：

```bash
docker compose -f compose.prod.yml stop app
docker run --rm -v zhugeshensuan_runtime-data:/data -v "$PWD/backups:/backup" alpine \
  cp /data/runtime.db /backup/runtime-$(date +%Y%m%d-%H%M%S).db
docker compose -f compose.prod.yml start app
```

恢复时先停止应用，将备份恢复为卷内 `/data/runtime.db`，再启动并检查 `/readyz`。恢复演练必须在非生产环境完成一次。

## 发布与回滚

发布前记录当前 `IMAGE_TAG`，构建新版本并完成真实模型、SSE、数据卷恢复测试。若新版本不就绪：

```bash
export IMAGE_TAG=4.1.0-previous
docker compose --env-file .env -f compose.prod.yml up -d --no-build
curl --fail http://127.0.0.1:8000/readyz
```

回滚只切换镜像版本，不回滚或删除 `runtime-data` 卷。若新版本改变运行库结构，必须先验证向后兼容或提供独立迁移与恢复步骤。

## 上线验收

1. CI 的 Black、Ruff、pytest、JavaScript 语法、pip-audit、镜像构建和容器健康检查全部通过。
2. 使用真实模型完成一次 POST SSE 论命，确认 Nginx/CDN 不缓冲且连接不中断。
3. 验证同一设备第 4 次每日请求返回 429，并检查 `Retry-After`。
4. 完成持久化卷备份恢复和上一版本镜像回滚。
