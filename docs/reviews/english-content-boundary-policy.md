# 英文产品内容边界与业主审查口径

> 对应任务：SAFE-001 / SAFE-002
> 确认日期：2026-07-15
> 最终审查人：项目所有者
> 文档状态：已固化；2026-07-15 已收紧为“只呈交极端判词和明确专业操作”

## 1. 目标

允许 Wise Oracle 正常谈论运势、传统象征和文化解释，只筛出特别极端、足以让产品明显表现得像医生、理财顾问、律师，或像掌握确定未来的判词。

本审查不是外部监管认证，也不是关键词合规测试，更不以防止任何理论上的误解或诉讼为目标。它采用普通用户在完整占卜页面中的合理理解，只把明确专业操作或极端结果判断交给项目所有者。

## 2. 不做什么

- 不以消除所有预测性语言为目标。
- 不因为出现 `health`、`wealth`、`will`、`should` 等单词就直接判定需要修改。
- 不因为出现 `investment`、`dividend`、`sector`、`heart`、`lung`、食物名称或一般症状词就直接判定问题。
- 不把奖金、分红、行业意象、一般财富趋势、储蓄、避免投机、普通饮食和散步等内容列为极端问题。
- 不自动改写签文、Prompt、页面或其他内容源。
- 不使用前端隐藏替代源内容审查。
- 不在项目所有者确认前建立关键词式 CI 内容门禁。
- 不把候选项、风险等级或执行者建议当成最终裁决。

## 3. 裁决权与职责

### 3.1 项目所有者

- 只对执行者已经完成语义初筛、并明确判断为存在风险的问题项作出“按建议修改 / 保留并接受风险 / 删除 / 另行指定”决定。
- 选择修改时，直接给出修改文本或修改方向；也可明确授权执行者先提出改写稿。
- 承担保留内容带来的产品和经营风险。
- 对执行后的内容差异进行第二次复核。
- 不负责替执行者检查全部 384 签，也不负责从高召回候选中辨别误报。

### 3.2 执行者

- 检查全部约定内容，先完成高召回收集，再结合上下文逐项作语义判断。
- 把“覆盖证明”“待分析候选”“一般边界表达”和“极端判词/明确专业操作”严格分开；只有最后一类进入项目所有者审查册。
- 对每个问题给出明确初筛结论、理由、完整上下文和非约束性修改建议。
- 对未列入问题清单的内容承担初筛责任，不把筛选工作转交项目所有者。
- 只实施已有明确业主决定的修改。
- 未决定项保持原文，不推断默许。

### 3.3 自动化或临时分析

- 只用于提高覆盖率、定位候选项和核对清单完整性。
- 可以基于字段、词组和句式扩大候选范围。
- 不输出“合规 / 违规”的最终结论。
- 不直接修改内容，也不因候选命中使 CI 失败。
- 自动化命中不得直接进入项目所有者审查册；必须经过执行者的语义判断。

## 4. 审查范围

### 4.1 英文签文

- 数据源：`data/content/oracle_signs_en.json`
- 覆盖：384/384 签
- 字段：`sign_text`、`interpretation1`、`career`、`wealth`、`love`、`health`、`study`、`general`
- 对照源：`data/content/oracle_signs_reinterpreted.json`
- 默认原则：原签诗和传统意象优先保留；`sign_text` 同时作为其他字段的必要上下文。

所有 `wealth` 和 `health` 字段必须由执行者检查，但不得因为字段名称天然敏感就全部进入业主审查册。普通行业意象、财富来源、分红、一般投资谨慎、普通养生和低风险生活提醒默认保留。只有严重或异常健康判断、治疗式操作、明确金融产品/仓位/买卖时机，或其他极端判词才进入问题清单。

### 4.2 英文黄历

- `data/content/huangli_terms_en.json`
- `data/content/huangli_scenarios_en.json`
- `zhugeshensuan/huangli_english.py` 中直接面向用户的固定英文提示
- `frontend/templates/en/daily_almanac.html`

检查全部字符串叶子和公开展示措辞，重点关注健康、财务、法律、灾难、确定性结果和重大人生决定。

### 4.3 英文文章与公开页面

- `frontend/templates/en/articles.html`
- `frontend/templates/en/article_detail.html`
- 英文 Disclaimer、Terms、About、Ask Oracle 和 Birth Chart 页面中与产品定位直接相关的文本

当前文章系统仅有占位内容。SAFE-002 记录现状和候选项；未来新增文章必须另行进入业主审查。

### 4.4 英文四柱 AI

- `prompts/birth_chart_en_prompt.md`
- `zhugeshensuan/birth_chart_english.py` 中的 fallback Prompt、固定 responsible-use 文本和输出解析边界
- `frontend/templates/en/birth_chart.html`

SAFE-002 可以完整审查仓库中现有的静态 Prompt 和固定文本，但不能穷举尚未由外部模型生成的未来输出。运行时输出属于持续风险，需要通过 Prompt 约束、抽样记录和项目所有者复核处理，不能宣称一次静态审查后永久完成。

## 5. 允许和默认保留

以下内容通常允许，不因其本身进入修改流程：

- 一般性的运势、吉凶趋势和可能性表达。
- 传统象征、卦象、宫位、五行、生肖、季节和文化背景。
- 使用 `may`、`might`、`could`、`suggests` 等表达的非确定性解释。
- 不涉及高风险具体操作的职业、关系、财富和健康主题。
- “保持耐心”“注意休息”“审慎考虑”“与他人沟通”等低风险生活提醒。
- 诗歌和原签中的死亡、疾病、灾难或财富意象，只要没有被扩写成对用户的现实诊断或指令。
- 清晰标明为传统观点、文化解释或自我反思的问题。

## 6. 必须由执行者完成语义判断的候选规则

候选规则用于扩大执行者的检查范围，不代表自动判定需要修改，也不代表必须交给项目所有者。执行者结合上下文确认“我认为确有风险”后，才呈交项目所有者作最终裁决。

| 代码 | 候选领域 | 呈交条件 |
| --- | --- | --- |
| MED-01 | 严重健康判断 | 从卦象直接推出急性疾病、严重或异常症状、心血管/神经系统风险等需要专业判断的结论；一般休息提醒和传统脏腑意象不算 |
| MED-02 | 治疗式健康操作 | 给出穴位、艾灸、针灸、手术、中医就诊、疗程或可能替代专业医疗的具体操作；普通食物、散步和早睡不算 |
| FIN-01 | 明确金融产品或资产 | 明确指向外汇、黄金、股票、债券、房地产、贵金属等可操作资产；奖金、分红、行业意象和一般财富来源不算 |
| FIN-02 | 明确交易或资金操作 | 给出买卖、止盈止损、仓位、DCA、投入比例、分批进场或具体投资时机；储蓄和“避免投机”等一般提醒不算 |
| LEG-01 | 法律行动或法律结果 | 建议起诉、签约、撤诉、报警或预测裁判、合同和纠纷结果 |
| LIFE-01 | 生育、死亡或严重灾难 | 预测怀孕、胎儿性别、不孕、死亡、重伤、事故、灾难或不可逆后果 |
| LIFE-02 | 重大人生决定 | 直接要求结婚、离婚、辞职、迁居、退学或作出类似高影响决定 |
| DET-01 | 保证或确定结果 | 对财富、健康、婚姻、考试、职业等给出保证、必然或无条件结果 |
| DET-02 | 精确结果时间 | 把具体月份、季节、年龄或期限写成用户现实结果会发生的时间 |
| MAN-01 | 恐吓和情绪操纵 | 用诅咒、灾祸、不可避免损害、紧迫感或羞辱推动用户行动 |
| PAY-01 | 诱导付费 | 暗示付款、升级或重复占卜能避免坏结果、提高准确性或解锁必要答案 |
| CTX-01 | 文化解释边界不清 | 可能是原典背景，也可能是后期模型增加的现代专业建议，需要业主结合上下文判断 |
| PROMPT-01 | 动态 AI 边界 | Prompt 或页面定位可能过严、过松或与本口径不一致，需要业主决定运行时策略 |

## 7. 问题初筛结论

只有达到“特别极端或明确专业操作”阈值的内容才能进入业主审查册。执行者必须明确写出：

- **初筛结论：** “我认为这是需要项目所有者裁决的极端判词/明确专业操作”。
- **判断理由：** 指明具体越过哪条边界，不再只写“可能涉及某主题”。
- **修改建议：** 说明应保留的文化内容与建议删除、概括或弱化的具体部分。

项目所有者仍保留最终决定权，可以选择修改、保留并接受风险、删除或另行指定方式。未被执行者判为问题的覆盖项不要求项目所有者逐项确认。

## 8. 审查册单项格式

每一问题项必须包含：

```markdown
## EXTREME-005-HEALTH

- 初筛结论：**我认为这是需要项目所有者裁决的极端判词/明确专业操作**
- 风险规则：MED-01
- 内容位置：data/content/oracle_signs_en.json / 第 5 签 / health
- 极端点：从卦象推出严重或异常健康结论

**签诗上下文**
> 完整英文 sign_text

**具体问题句**
> 逐句引用达到极端阈值的英文原句

**当前英文原文**
> 完整字段原文

**我的非约束性修改建议**
> 保留文化解释，删除或概括具体专业建议

**项目所有者决定**
- [ ] 按建议方向修改
- [ ] 保留原文并接受风险
- [ ] 删除该字段
- [ ] 另行指定修改方式

**项目所有者修改要求或目标文本**

> 待填写

**复核状态**
- [ ] 已作决定
- [ ] 已实施
- [ ] 已复核
```

## 9. 完整性标准

SAFE-002 只有同时满足以下条件才算完成问题审查册：

1. 384 个签号连续且无重复、无缺失，8 个公开文本字段均由执行者检查。
2. 384 个 `wealth` 和 384 个 `health` 字段全部计入覆盖证明，但只有经语义判断达到极端阈值的字段进入业主审查册。
3. 其余六个文本字段（包括 `sign_text`）均已执行收集与语义判断，并记录覆盖计数。
4. 每个进入业主审查册的问题都包含明确初筛结论、逐句的具体问题句、完整英文原文和签诗上下文。
5. 原签诗、传统意象、一般运势和低风险提醒不因关键词命中被误列为问题。
6. 英文黄历两个 JSON 的全部字符串叶子均已检查并记录数量。
7. 文章占位页、相关英文定位页面、四柱静态 Prompt 和 fallback 文本均列入范围证明。
8. 审查册总索引列出所有分册、问题数、规则分布和待审数量；覆盖证明单独保存，不转化为业主待办。
9. 所有项目所有者决定栏初始均为“待填写”。
10. SAFE-002 不修改任何受审内容源。

## 10. 审查和实施流程

1. 执行者完成全量覆盖、高召回收集和语义初筛，生成问题清单与独立覆盖证明。
2. 项目所有者只对问题清单逐项选择按建议修改、保留并接受风险、删除或另行指定方式。
3. 项目所有者选择修改时，给出文本或明确的修改要求。
4. 执行者把决定转录到结构化决定记录，等待确认。
5. 执行者只修改已经确认的项目。
6. 项目所有者检查修改前后差异并完成复核。
7. 仅在项目所有者明确指定硬红线后，考虑建立最小定向回归保护。

## 11. 当前限制

- 极端阈值仍可能存在判断分歧；项目所有者可以推翻执行者的修改建议并明确选择保留。
- 语言风险依赖上下文，任何自动收集都不能证明没有遗漏全部语义风险。
- 外部 AI 输出具有非确定性；静态仓库审查不能覆盖未来所有模型响应。
- 本文档不是特定司法辖区的法律意见，外部平台和监管要求仍需在实际上线、投放或支付接入时实时核验。
