## 字符串分类说明

| 分类 | 说明 | i18n 处理建议 |
|------|------|--------------|
| **UI 文本** | 用户可见的界面文字（标题、按钮、标签等） | 必须抽取为 i18n 字符串 |
| **状态提示** | 操作过程中的临时状态信息（加载中、成功、失败等） | 必须抽取为 i18n 字符串 |
| **错误消息** | 异常情况下展示给用户的信息 | 必须抽取为 i18n 字符串 |
| **数据字典** | 时辰名称、天干地支、数字汉字等传统文化数据 | 需单独评估，部分可能保持中文或建立多语言映射 |
| **调试日志** | `console.log` / `console.error` 中的中文 | 可不 i18n，仅开发可见；建议保留中文或改为英文 |
| **占位符/模板** | 含变量的模板字符串 | 抽取时保留变量占位符 |

## 三、i18n 处理建议

### 3.1 优先级分级

| 优先级 | 分类 | 处理时机 | 说明 |
|--------|------|---------|------|
| **P1-必须** | UI 文本、状态提示、错误消息 | 繁体化阶段 | 用户直接可见，必须翻译 |
| **P1-必须** | HTML 模板静态文本 | 繁体化阶段 | 页面主体内容 |
| **P2-评估** | 数据字典（时辰、天干地支、数字汉字） | 繁体化阶段评估 | 传统文化术语，繁体可能保持原样或需映射 |
| **P3-可选** | 调试日志 | 英文阶段或不动 | 仅开发可见，可保留中文或改英文 |

### 3.2 数据字典特殊处理建议

以下内容属于中国传统文化固定术语，建议单独建立多语言映射表，不混入普通字符串表：

1. **天干地支**：`甲乙丙丁戊己庚辛壬癸` / `子丑寅卯辰巳午未申酉戌亥`
   - 繁体：保持原样
   - 英文：可音译（JiaYiBing...）或意译，需产品决策

2. **时辰名称**：`子时` `丑时` ... + 描述 `夜半` `鸡鸣` ...
   - 繁体：保持原样
   - 英文：需产品决策（音译 / 时段描述 / 双语）

3. **中文数字**：`〇一二三四五六七八九十` / `初一初二...三十`
   - 繁体：保持原样
   - 英文：建议改为阿拉伯数字或英文序数

4. **农历月份**：`正月` `腊月` `冬月` `十月` / `闰`
   - 繁体：保持原样
   - 英文：需产品决策

5. **场景映射**：`结婚→婚嫁` `搬家→搬迁` `开业→开市`
   - 繁体：`结婚→結婚` `搬家→搬家` `开业→開業`（显示名 `婚嫁` `搬迁` `开市` 繁体为 `婚嫁` `搬遷` `開市`）
   - 英文：需重新映射

### 3.3 重复字符串去重建议

以下字符串在多个文件/位置重复出现，建议在 i18n 字符串表中定义为单一键：

| 字符串 | 出现位置 | 建议键名 |
|--------|---------|---------|
| `加载中...` | huangli.html (19 处) / huangli.js | `common.loading` |
| `无` | huangli.js (10+ 处) | `common.none` |
| `未知` | huangli.js / lunming.js / lunar_date_handler.js | `common.unknown` |
| `网络响应不正常` | huangli.js (2 处) | `error.network` |
| `传统文化娱乐参考，不构成医疗、投资或人生决策建议。` | suanshi/huangli/lunming.html | `common.disclaimer` |
| 诗歌（算事页 8 行） | index.html / suanshi.html | `poem.suanshi.*` |
| 介绍文字（算事页 9 段） | index.html / suanshi.html | `intro.suanshi.*` |
| `请书三字` / `开始测算` / `笔画数` 等 | index.html / suanshi.html | `suanshi.*` |

### 3.4 模板字符串处理

含变量的模板字符串需在 i18n 方案中支持占位符替换：

| 原始字符串 | 建议格式 |
|-----------|---------|
| `当前时间：${formattedTime} · ${chineseHour}` | `current_time: "当前时间：{time} · {hour}"` |
| `所选日期子时：${data.gan_zhi_hour}` | `selected_date_zi_hour: "所选日期子时：{hour}"` |
| `(${days}天前)` | `days_ago: "({days}天前)"` |
| `(还有${days}天)` | `days_later: "(还有{days}天)"` |
| `${month}月${day}日 ${weekday}` | `date_short: "{month}月{day}日 {weekday}"` |
| `修正 ${minutes} 分钟` | `correction_minutes: "修正 {minutes} 分钟"` |
| `${type}解签` | `interpretation_type: "{type}解签"` |

## 四、待确认事项与决策记录

以下问题在 P0 审计期间经讨论已明确决策，作为 P1 执行依据。

### 4.1 已决策事项

#### 决策 1：lunar.js 原生 i18n 能力（已确认）

**结论**：本地 [lunar.js#L8374-L8467](file:///v:/诸葛神算V4/frontend/static/js/lunar.js#L8374-L8467) 已内置 I18n 模块，对外暴露四个方法：
- `I18n.setLanguage(lang)` — 切换语言（原生支持 `chs` 简体、`en` 英文）
- `I18n.getLanguage()` — 获取当前语言
- `I18n.setMessages(lang, messages)` — 自定义/更新语言配置表
- `I18n.getMessage(key)` — 按 key 取当前语言字符串

**影响**：黄历核心数据（干支、节气、生肖、节日名等）的翻译工作量大幅降低。繁体化可通过 `I18n.setMessages('zh-hant', {...})` 注入繁体配置；英文阶段直接 `I18n.setLanguage('en')`。

**注意**：官网提示多语言为实验特性，需在 P1 实测验证 `setMessages` 注入繁体的可行性。

#### 决策 2：诗歌与介绍文字策略（已确认）

**结论**：
- **繁体阶段（P1）**：简繁字体替换，无需重写
- **英文阶段（P5）**：诗歌重写、介绍文字翻译/重写

**影响**：P1 阶段这两块只需做字符级转换，不涉及内容创作。

#### 决策 3：统一多语言字典（已确认采纳）

**结论**：建立统一多语言字典，避免同一概念在多处重复维护。

**典型场景**：`结婚/搬家/开业` 等场景词出现在 3 处——`huangli.html` 的 `data-scenario` 属性值、按钮显示文本、`huangli.js` 的 `scenarioNames` 映射对象。统一字典后只需维护一处。

**影响**：P1 阶段需设计字典结构与键名命名规范，建立初始映射表。

#### 决策 4：时区处理（已确认保留）

**结论**：时区字段保留，参与计算。

**命理依据**：传统命理以出生地当地时间为准。例如美国人中午 12 点出生应记为"午时"，不应换算到中国时区变成晚上。时区字段用于锚定出生地的标准时间，是正确且必要的。

**P1 阶段处理**：时区下拉选项显示文本繁体化（`中国标准时间`→`中國標準時間` 等），`UTC` 保持不变。

**P5 待办**：当前选项仅 5 个（前 4 个均为 UTC+8，仅 `UTC` 与其他有实质差异），无法支持海外用户。P5 英文商业化阶段需扩展为全球时区选择器。

#### 决策 5：真太阳时功能（已确认保留，默认关闭）

**结论**：保留真太阳时功能，默认关闭，作为可选项。

**命理依据**（将同步记入 FAQ 页面）：
- 古代用日晷/漏刻计时，日晷测量的就是真太阳时
- 现代钟表时间是平太阳时（北京时间=东经120°平太阳时）
- 八字体系建立在真太阳时基础上，清代袁树珊《命理探源》明确支持
- 系统提供真太阳时选项供专业用户选用，默认关闭以避免普通用户困惑

**影响**：
- P1 阶段：真太阳时相关 UI 文本繁体化
- FAQ 页面需记录真太阳时问题（古代计时方式、真太阳时与平太阳时区别、为何设为可选项）

#### 决策 6：出生地输入与经度获取（已确认）

**结论**：采用方案 C——始终要求输入出生地（文本），系统自动获取经度+时区。

**技术方案**：引入 Nominatim（地名→经纬度）+ timezonefinder（经纬度→时区，Python 离线库）。全免费，无 key。

**用户交互**：
1. 用户输入出生时间 + 出生地（文本，如"成都"）
2. 系统自动调用 Nominatim 获取经度，调用 timezonefinder 获取时区
3. 默认不勾选"真太阳时"，用标准时区排盘
4. 用户可勾选"真太阳时"，系统用已获取的经度自动计算
5. **用户全程不接触经度概念**

**未采纳的优化**：曾讨论"非争议区域跳过查询"的分区策略，但因判断争议区域本身需要经度（鸡生蛋问题），且收益有限（Nominatim 免费），按"避免过度工程"原则不做此优化，统一查询。
