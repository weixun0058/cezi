# P1 繁体改造 — 技术难点分析报告

**分析日期：** 2026-06-24

**分析对象：** V:/诸葛神算V4，P1 繁体中文改造阶段（zh-Hant）

---

## 难点 1（结构性）：模板完全独立，无共享骨架

四个简体模板（`index.html`、`suanshi.html`、`huangli.html`、`lunming.html`）都是独立 HTML 文件，`<head>` 引入的 JS/CSS、nav-links、poetry-section、scroll-container/scroll-header/scroll-content/scroll-footer 层层嵌套的结构完全手工复制。

计划要求之间存在矛盾：
- "新建：`frontend/templates/base.html`"
- "新建繁体模板"
- "不改动原有简体模板，除非共享布局确实需要"

如果 `base.html` 只给繁体用、不动简版，风险可控。但如果真想抽取共享骨架，就必须一次性重构所有四个简版模板——超出了 P1 的边界，可能因为某个 JS 选择器或 class 名在重构中损坏而导致回流。

**建议：** P1 直接硬复制简版模板做繁体版、手动改文案，不做 base.html 提取。base.html 放到 P5 英文阶段（12 个模板）再做，收益才够。

## 难点 2（工作量最大）：JS 中全部硬编码中文交互文案

三个核心 JS 文件里有大量硬编码中文，需要覆盖替换：

| 文件 | 硬编码示例 |
|------|-----------|
| main.js | "正在查询笔画…"、"笔画查询失败，请重试"、"请选择求测方向"、"无效的签号数据" |
| huangli.js | "当前时间："、"加载中..."、"未知"、"冲煞: "、"喜神: "、"神煞: " |
| lunming.js | "请输入姓名"、"请选择出生日期"、"命盘纲要"、"四柱排盘"、"年柱"、"月柱"、"正在分析，请稍候..." |
| huangli.js 时辰表 | "子时"、"丑时"…"亥时" 全部 12 时辰名称及描述（夜半、鸡鸣…） |
| huangli.js 星期对照 | ['周日', '周一', '周二', …, '周六'] |
| huangli.js 场景标签 | { '结婚': '婚嫁', '搬家': '搬迁', '开业': '开市' } |
| huangli.js 月份/日期组合 | `${month}月${day}日 ${weekday}` |

当前 CSP 为 `script-src 'self'`，阻止 inline script。可行的客户端 i18n 方案：

- **方案 A**：繁体页面额外加载一个 `lang-zh-hant.js`（只定义字符串表或 `WISE_ORACLE_LANG = 'zh-hant'`），与 CSP 兼容
- **方案 B**：繁体版各加载独立的 JS 副本（如 `main_zh_hant.js`），安全但维护成本翻倍
- **方案 C**：JS 内做运行时语言检测 + 集中式 strings 对象，单个 JS 文件内切换

## 难点 3（设计矛盾）：后端 API 错误消息为简体中文

计划 P1.3 规定：
- "不翻译后端错误码"
- "在前端增加繁体加载、空状态、错误、重试、结果、校验提示"

后端大量错误消息是简体中文，例如：

- divination.py: "请求体必须是 JSON 对象"、"请输入一个汉字"、"暂时无法查询该字笔画"
- huangli_api.py: "日期格式无效"、"不支持的场景筛选"
- lunming_api.py: "请求体必须是 JSON 对象"、"姓名不能为空"、"请提供出生日期"

前端 JS catch 块直接展示 `data.error?.message`，繁体用户会看到简体消息。

**可行方案：** JS 做 code-based 映射——用 `error.code`（如 `INVALID_CHARACTER`）查繁体消息表，忽略 `error.message`。但当前 JS 是 `throw new Error(data.error?.message)`，需要修改错误处理逻辑的走向。

## 难点 4（已验证无风险）：笔画查询与繁体字输入

**初版判断已撤销。** 经 DB 实测确认，笔画查询完全支持繁体字输入，不存在核心流程中断风险。

database.py 的 get_stroke_count 执行：
```sql
SELECT simplified_strokes, traditional_strokes, kangxi_strokes
FROM hanzi WHERE character = ?
```

`WHERE character = ?` 用用户输入的字符去查。**实测确认 `hanzi` 表同时存储简体字和繁体字作为独立行**，且查询逻辑优先返回 `kangxi_strokes`，康熙笔画对简繁统一。

**实测数据（2026-06-24）：**

| 输入 | simplified | traditional | kangxi | 查询结果 |
|------|-----------|-------------|--------|---------|
| `龙`（简） | 5 | 16 | 16 | 命中，返回 16 |
| `龍`（繁） | 16 | 0 | 16 | 命中，返回 16 |
| `马`（简） | 3 | 10 | 10 | 命中，返回 10 |
| `馬`（繁） | 10 | 0 | 10 | 命中，返回 10 |
| `书`（简，无繁体对应） | 4 | 0 | 10 | 命中，返回 10 |

**关键事实：**
1. `hanzi` 表共 18620 条记录，其中 7456 条有 traditional_strokes。
2. 简体字"龙"和繁体字"龍"在表里是独立两行，都能查到。
3. 查询逻辑优先返回 `kangxi_strokes`，康熙笔画对简繁统一。
4. 诸葛神算系统本就以繁体字（康熙字典笔画）为依托，简体字输入时查的也是康熙笔画。繁体字输入反而更贴合系统本源。
5. 仅当输入字是后造简体字、在繁体字库中无对应时，才采用其本身的简体笔画。

**结论：** 繁体字输入不是问题。此难点从"高风险"降级为"无风险"。

## 难点 5（易遗漏）：场景过滤的数据键值是中文字段

huangli.js 九日黄历展示：
```javascript
const scenarioNames = { '结婚': '婚嫁', '搬家': '搬迁', '开业': '开市' };
```

模板按钮：
```html
<button data-scenario="结婚">婚嫁</button>
```

`data-scenario` 的值发送到后端 `/api/week_huangli?scenario=结婚`，和后端 SCENARIOS 字典绑定：
```python
SCENARIOS = { "结婚": ("嫁娶", "结婚"), "搬家": ("移徙", "入宅", "搬家"), ... }
```

所以 `data-scenario` 的值不能改，模板按钮显示文本和 JS 里的 `scenarioNames` 表需要做繁体适配。

## 难点 6（低风险）：时间显示中的中文字段

huangli.js 的 getChineseHour 返回格式："子时 (夜半, 23:00-1:00)" — 含时辰名和描述两种中文。

updateCurrentTime() 输出："当前时间：14:30 · 未时 (日昳, 13:00-15:00)"

displayHuangliData 使用 `toLocaleDateString('zh-CN', ...)` 格式化日期。繁体版应改用 `'zh-Hant'` locale。
