# 部署指南

本文档是本项目从一台空白 Ubuntu 服务器部署到可公网访问生产服务的完整方案。当前推荐部署方式是：本地构建 Docker 镜像，上传到服务器，服务器导入镜像后通过 Docker Compose 运行，公网入口由 Cloudflare Tunnel 负责。

没有域名、只想临时用 IP 验证服务时，见 `docs/no-domain-access-notes.md`。临时 IP 访问不是正式上线方案。

## 目标架构

```text
用户浏览器
  -> https://你的域名/
  -> Cloudflare
  -> Cloudflare Tunnel
  -> 127.0.0.1:8000
  -> Docker 容器 zhugeshensuan:local
```

关键约束：

- Docker 容器只绑定服务器本机 `127.0.0.1:8000`。
- 服务器不需要直接开放 `80` 和 `443` 给源站应用；公网入口由 Cloudflare 提供。
- 不直接把 Docker 的 `8000` 端口暴露到公网。
- 服务器运行项目时不需要手动执行 `python app.py`。
- `.env` 只放在部署目录，不提交仓库，不写进镜像，不写进 Nginx 配置。

## 准备信息

部署前准备：

- 一台 Ubuntu 服务器，本文以搬瓦工 Ubuntu 为例。
- 服务器 SSH 登录信息，例如 `root@服务器IP`。
- 一个 Cloudflare 托管的域名；当前实测域名为 `getwiseoracle.com`。
- 本地 Windows 项目目录：`V:\诸葛神算V4`。
- 本地 Docker 镜像：`zhugeshensuan:local`。

使用 Cloudflare Tunnel 时，不需要把域名 A 记录指向服务器公网 IP。Tunnel 创建 Published application route 后，Cloudflare 会自动配置对应 DNS 路由。

## 一、本地准备部署文件

在本地 Windows PowerShell 执行：

```powershell
cd "V:\诸葛神算V4"
docker build -f deploy/Dockerfile --build-arg APP_VERSION=local -t zhugeshensuan:local .
docker images | Select-String "zhugeshensuan"
```

应能看到：

```text
zhugeshensuan:local
```

导出镜像：

```powershell
docker save zhugeshensuan:local -o zhugeshensuan-local.tar
```

需要上传到服务器的文件：

```text
zhugeshensuan-local.tar
deploy/compose.prod.yml（上传后命名为 compose.prod.yml）
.env
```

其中 `.env` 必须包含：

```env
IMAGE_NAME=zhugeshensuan
IMAGE_TAG=local
APP_PORT=8000
APP_ENV=production
APP_DEBUG=false
SITE_BASE_URL=https://getwiseoracle.com
CONTACT_EMAIL=5siwei@gmail.com
ARTICLE_ADMIN_PATH=165131
ARTICLE_ADMIN_PASSWORD=设置一个只由站长知道的访问密码
TRUSTED_PROXY_HOPS=1
SECRET_KEY=替换为生产密钥
AI_API_KEY=替换为模型供应商密钥
AI_BASE_URL=https://api.deepseek.com
AI_MODEL=deepseek-v4-flash
AI_GLOBAL_DAILY_LIMIT=100
```

`SITE_BASE_URL` 是 canonical、hreflang、Open Graph、JSON-LD、sitemap 和 robots 使用的唯一生产 origin。生产值必须是公开 HTTPS 根地址，不带路径、查询参数或末尾斜杠；应用不会根据请求 `Host` 或代理转发的 host 动态生成 SEO URL。测试/预发布环境必须显式使用各自的 `SITE_BASE_URL`，不得让预发布域名写入正式页面源码。

`CONTACT_EMAIL` 是 Contact、Privacy、Terms 和 Disclaimer 统一公开的联系邮箱。生产环境必须显式配置有效的公开邮箱；空值、无效格式和 `.example` 占位域名都会导致应用拒绝启动。

`ARTICLE_ADMIN_PATH=165131` 形成私密文章入口 `https://getwiseoracle.com/165131`。它必须是 6 位数字；以后修改该环境变量并重启容器即可让旧 URL 失效。`ARTICLE_ADMIN_PASSWORD` 是这个页面的简单访问密码，建议配置；留空时只依赖 URL 后缀，上传功能仍可用。

`docker compose --env-file .env` 只为 Compose 提供变量替换；`deploy/compose.prod.yml` 已在容器 `environment` 中显式传递公开配置、文章入口配置，并把文章目录固定为持久卷内的 `/app/content/articles_en`。

不要把 `.env` 发到 GitHub，也不要截图外发。

## 二、初始化 Ubuntu 服务器

登录服务器：

```powershell
ssh root@服务器IP
```

更新系统并安装基础工具：

```bash
sudo apt update
sudo apt upgrade -y
sudo apt install -y ca-certificates curl gnupg ufw
```

基础防火墙只需要保留 SSH。Cloudflare Tunnel 通过服务器主动出站连接 Cloudflare，不需要服务器入站开放 `80/443`。

```bash
sudo ufw allow OpenSSH
sudo ufw enable
sudo ufw status
```

如果后续改用传统 Nginx/Certbot 方案，再开放 `80/443`。

## 三、安装 Docker

使用 Docker 官方 apt 仓库安装 Docker Engine 和 Compose v2：

```bash
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

验证：

```bash
sudo docker version
sudo docker compose version
sudo docker run hello-world
```

可选：允许当前用户不加 `sudo` 执行 Docker。执行后需要退出 SSH 并重新登录。

```bash
sudo usermod -aG docker $USER
```

在服务器执行：

```bash

```

必须看到：

```text

```

检查 `.env` 是否会让 Compose 使用正确镜像：

```bash
grep -E '^(IMAGE_NAME|IMAGE_TAG|APP_PORT|SITE_BASE_URL|CONTACT_EMAIL|ARTICLE_ADMIN_PATH|AI_BASE_URL|AI_MODEL|AI_GLOBAL_DAILY_LIMIT)=' .env
```

必须包含：

```env
IMAGE_NAME=zhugeshensuan
IMAGE_TAG=local
APP_PORT=8000
SITE_BASE_URL=https://getwiseoracle.com
CONTACT_EMAIL=5siwei@gmail.com
ARTICLE_ADMIN_PATH=165131
AI_BASE_URL=https://api.deepseek.com
AI_MODEL=deepseek-v4-flash
AI_GLOBAL_DAILY_LIMIT=100
```

启动容器。这里必须使用 `--no-build`，因为服务器上只上传了镜像和 Compose 配置，没有上传完整源码；同时使用 `--force-recreate`，确保同一标签重新导入镜像后不会继续运行旧容器。该参数不会删除持久卷：

```bash
docker compose --env-file .env -f compose.prod.yml up -d --no-build --force-recreate
```

检查状态：

```bash
docker compose --env-file .env -f compose.prod.yml ps
curl --fail http://127.0.0.1:8000/healthz
curl --fail http://127.0.0.1:8000/readyz
```

`/healthz` 只表示进程存活；`/readyz` 同时验证生产配置和 SQLite 可用性。只有 `/readyz` 正常后，才继续配置公网入口。

## 六、配置 Cloudflare Tunnel

当前项目源站容器只监听服务器本机 `127.0.0.1:8000`。Cloudflare Tunnel 会把公网域名请求转发到这个本机地址，不需要暴露源站 IP。

### 1. 进入 Cloudflare Zero Trust

进入：

```text
https://one.dash.cloudflare.com/
```

如果第一次启用 Zero Trust：

- 选择 Free 计划。
- 点击 `Continue to Zero Trust`。
- 如果跳转 404，手动重新打开 `https://one.dash.cloudflare.com/`。

进入后依次打开：

```text
Networks -> Connectors -> Add a tunnel
```

Tunnel 类型选择：

```text
Cloudflared
```

Tunnel 名称建议：

```text
zhugeshensuan
```

操作系统选择：

```text
Debian
64-bit
```

Ubuntu 属于 Debian 系，Cloudflare 页面没有 Ubuntu 时选择 Debian。

### 2. 安装 cloudflared

在服务器执行 Cloudflare 页面给出的 Debian 安装命令。当前可用命令如下：

```bash
sudo mkdir -p --mode=0755 /usr/share/keyrings
curl -fsSL https://pkg.cloudflare.com/cloudflare-public-v2.gpg | sudo tee /usr/share/keyrings/cloudflare-public-v2.gpg >/dev/null
echo 'deb [signed-by=/usr/share/keyrings/cloudflare-public-v2.gpg] https://pkg.cloudflare.com/cloudflared any main' | sudo tee /etc/apt/sources.list.d/cloudflared.list
sudo apt-get update && sudo apt-get install cloudflared
```

验证：

```bash
cloudflared --version
```

本次实测版本：

```text
cloudflared version 2026.6.1
```

### 3. 安装 Tunnel 系统服务

复制 Cloudflare 页面给出的第一条命令：

```bash
sudo cloudflared service install <页面给出的token>
```

不要使用只在当前终端临时运行的命令：

```bash
cloudflared tunnel run --token <token>
```

检查服务：

```bash
sudo systemctl status cloudflared --no-pager
sudo journalctl -u cloudflared -n 30 --no-pager
```

`journalctl` 在部分系统上不支持 `--tail=30`，用 `-n 30`。

Cloudflare 控制台中 Tunnel 状态应为：

```text
HEALTHY
```

### 4. 添加 Published application route

在 Tunnel 详情页打开：

```text
Published application routes -> Add a published application route
```

根域名访问，例如 `https://getwiseoracle.com/`，填写：

```text
Subdomain: 留空
Domain: getwiseoracle.com
Path: 留空
Service Type: HTTP
URL: 127.0.0.1:8000
```

注意：

- `Subdomain` 不要填 `https://getwiseoracle.com`。
- 页面应显示 `Full hostname: getwiseoracle.com`。
- `Type` 选择 `HTTP` 后，`URL` 只填 `127.0.0.1:8000`，不要再写 `http://`。

保存后，浏览器访问：

```text
https://getwiseoracle.com/
```

本次已验证：`https://getwiseoracle.com/` 能打开项目页面。

### 5. 可选：开启 HTTPS 强制跳转

Cloudflare Tunnel 发布后，`https://getwiseoracle.com/` 通常已经可用。如果 `http://getwiseoracle.com/` 也能打开，建议开启 HTTP 到 HTTPS 的自动跳转。

在 Cloudflare 主控制台，不是 Zero Trust 控制台，进入：

```text
getwiseoracle.com -> SSL/TLS -> Edge Certificates
```

开启：

```text
Always Use HTTPS
```

开启后：

```text
http://getwiseoracle.com/
```

会自动跳转到：

```text
https://getwiseoracle.com/
```

### 6. 可选：让 www 域名也能访问

如果希望 `www.getwiseoracle.com` 也能访问同一个网站，先在 Cloudflare Tunnel 中再添加一条 Published application route。

路径：

```text
Zero Trust -> Networks -> Connectors -> zhugeshensuan -> Published application routes -> Add a published application route
```

填写：

```text
Subdomain: www
Domain: getwiseoracle.com
Path: 留空
Service Type: HTTP
URL: 127.0.0.1:8000
```

保存后，等待 DNS 生效。可用公共 DNS 检查：

```powershell
nslookup -type=A www.getwiseoracle.com 1.1.1.1
nslookup -type=AAAA www.getwiseoracle.com 1.1.1.1
```

如果公共 DNS 已经能解析，但本机浏览器仍不能访问，先刷新本机 DNS 缓存：

```powershell
ipconfig /flushdns
```

然后重新打开浏览器或使用无痕窗口访问：

```text
https://www.getwiseoracle.com/
```

### 7. 可选：把 www 永久跳转到裸域名

从 SEO 和维护角度，建议只选择一个主域名。当前建议使用裸域名作为主域名：

```text
https://getwiseoracle.com/
```

并把 `www` 永久跳转到裸域名：

```text
https://www.getwiseoracle.com/* -> https://getwiseoracle.com/*
```

好处：

- 避免 `www` 和裸域名形成重复页面。
- 集中链接权重和索引信号。
- sitemap、Search Console、统计数据和站内链接更统一。

在当前 Cloudflare 界面中，可使用 Page Rules 配置。

路径：

```text
getwiseoracle.com -> Rules -> Page Rules -> Create Page Rule
```

填写：

```text
URL: www.getwiseoracle.com/*
Setting: Forwarding URL
Status Code: 301 - Permanent Redirect
Destination URL: https://getwiseoracle.com/$1
```

不要再添加 `getwiseoracle.com/* -> ...` 的跳转规则，否则容易造成循环跳转。

`http -> https` 已由 `Always Use HTTPS` 负责，不需要额外写 Page Rule。

保存后测试：

```powershell
curl.exe -I https://www.getwiseoracle.com/
```

应看到类似：

```text
HTTP/2 301
location: https://getwiseoracle.com/
```

## 八、网站升级发布流程

功能添加、修改、修复都先在本地完成。Cloudflare Tunnel 不需要重新配置；它只负责把 `getwiseoracle.com` 转发到服务器本机 `127.0.0.1:8000`。只要新容器仍然监听这个地址，Tunnel 会继续生效。

**日常发表文章不属于网站升级。** 首次部署包含文章管理功能的新镜像后，站长直接访问 `https://getwiseoracle.com/165131` 上传 Markdown；上传后立即生效，不需要 GitHub、重新构建镜像或执行本节部署流程。只有程序代码、模板或依赖发生变化时才按下面流程升级容器。

推荐升级流程：

```text
本地改代码
-> 本地测试
-> 本地重新构建 Docker 镜像
-> 导出镜像 tar
-> 上传服务器
-> 服务器 docker load 导入
-> docker compose 重启应用容器
-> 验证本机 readyz 和公网域名
```

### 1. 本地测试

在本地完成代码修改后，至少运行与改动相关的检查。常用检查命令：

```powershell

.\.venv\Scripts\python.exe -m pytest -W error::ResourceWarning
.\.venv\Scripts\python.exe -c "import pathlib, subprocess; files=sorted(pathlib.Path('frontend/static/js').rglob('*.js')); assert files; [subprocess.run(['node', '--check', str(path)], check=True) for path in files]; print(f'node --check: {len(files)}/{len(files)} passed')"
```

```powershell

.\.venv\Scripts\python.exe -m ruff check app.py zhugeshensuan deploy/gunicorn.conf.py scripts tests
.\.venv\Scripts\python.exe -m pip_audit -r requirements.txt --no-deps --progress-spinner off
```

### 6. 哪些改动需要额外处理

只改 Python、HTML、CSS、JavaScript，一般只需要重新构建并替换镜像。

如果改了 `.env` 配置项，需要同步修改服务器上的 `.env`，然后重启容器：

```bash
docker compose --env-file .env -f compose.prod.yml up -d --no-build
```

如果改了数据库结构、运行时数据格式或 `instance/runtime.db` 的读写逻辑，不能只替换镜像。必须先备份数据卷，并准备迁移和回滚方案。

> **数据源说明（2026-07-13 确立）**：签文和彭祖百忌已从 `reference.db` 迁移为 JSON 内存加载，`reference.db` 运行时仅用于汉字笔画查询；其中的旧内容表只为构建可重现性保留。改签文只需改 JSON + 跑 `scripts/build_hant_json.py` + 重启，无需 backfill 到数据库。详见 `docs/architecture/data-source-migration-2026-07-13.md`。

如果改了 Docker 端口、Compose 服务名或容器监听地址，需要同时检查 Cloudflare Tunnel 的 Published application route 是否仍然指向：

```text
HTTP -> 127.0.0.1:8000
```

### 7. 回滚

使用版本号标签时，回滚只需要把服务器 `.env` 的 `IMAGE_TAG` 改回上一版，然后重启：

```bash
cd ~/zhugeshensuan
nano .env
docker compose --env-file .env -f compose.prod.yml up -d --no-build
curl --fail http://127.0.0.1:8000/readyz
curl --fail https://getwiseoracle.com/readyz
```

回滚只切换镜像版本，不要删除 `zhugeshensuan_runtime-data` 或 `zhugeshensuan_article-content` 数据卷。文章与程序镜像解耦，回滚代码不会撤回已经上传的文章。

## 九、备份与恢复

运行时数据库在 Docker 卷 `zhugeshensuan_runtime-data` 中，对应容器内 `/app/instance`；上传的 Markdown 文章在 `zhugeshensuan_article-content` 中，对应 `/app/content/articles_en`。两者都必须备份。管理页面的“打包下载全部文章”可随时导出全部 `.md`，但不代替服务器卷备份。

停机备份：

```bash
cd ~/zhugeshensuan
mkdir -p backups
docker compose --env-file .env -f compose.prod.yml stop app
docker run --rm -v zhugeshensuan_runtime-data:/data -v "$PWD/backups:/backup" alpine \
  cp /data/runtime.db /backup/runtime-$(date +%Y%m%d-%H%M%S).db
docker run --rm -v zhugeshensuan_article-content:/data:ro -v "$PWD/backups:/backup" alpine \
  tar -czf /backup/articles-$(date +%Y%m%d-%H%M%S).tar.gz -C /data .
docker compose --env-file .env -f compose.prod.yml start app
curl --fail http://127.0.0.1:8000/readyz
```

恢复时先停止应用：数据库备份恢复为 runtime 卷内 `/data/runtime.db`；文章压缩包解压到 article-content 卷根目录并确认存在 `/data/articles_en/*.md`。随后启动应用，检查 `/readyz`、`/articles` 和一篇文章详情。恢复演练必须先在非生产环境完成。

## 常见问题与处理

### 1. `failed to read dockerfile: open Dockerfile: no such file or directory`

现象：

```text
Image zhugeshensuan:4.1 Building
failed to solve: failed to read dockerfile: open Dockerfile: no such file or directory
```

原因：

- 服务器上只上传了镜像、`.env` 和 `compose.prod.yml`，没有上传完整源码和 `Dockerfile`。
- Compose 正在找默认镜像 `zhugeshensuan:4.1`，不是已导入的 `zhugeshensuan:local`。
- 通常是服务器 `.env` 缺少 `IMAGE_TAG=local`，或启动命令没有带 `--env-file .env`。

处理：

```bash
cd ~/zhugeshensuan
docker load -i zhugeshensuan-local.tar
docker images | grep zhugeshensuan
grep -E '^(IMAGE_NAME|IMAGE_TAG|APP_PORT|AI_MODEL)=' .env
docker compose --env-file .env -f compose.prod.yml up -d --no-build
```

`.env` 必须包含：

```env
IMAGE_NAME=zhugeshensuan
IMAGE_TAG=local
```

### 2. `docker compose ps` 没有容器

现象：

```text
NAME      IMAGE     COMMAND   SERVICE   CREATED   STATUS    PORTS
```

原因：

- 容器没有成功创建。
- 前一步 `up` 失败。

处理：

```bash
cd ~/zhugeshensuan
docker compose --env-file .env -f compose.prod.yml up -d --no-build
docker compose --env-file .env -f compose.prod.yml logs --tail=100 app
docker compose --env-file .env -f compose.prod.yml ps
```

### 3. `curl 127.0.0.1:8000/readyz` 失败

现象：

```text
curl: (7) Failed to connect to 127.0.0.1 port 8000
```

原因：

- 容器未启动。
- 容器启动后健康检查失败。
- 端口没有绑定到服务器本机。

处理：

```bash
docker compose --env-file .env -f compose.prod.yml ps
docker compose --env-file .env -f compose.prod.yml logs --tail=100 app
sudo ss -lntp | grep ':8000'
```

### 4. 浏览器访问 `http://服务器IP:8000` 失败

这是当前生产配置下的预期行为。

`compose.prod.yml` 只绑定：

```text
127.0.0.1:8000->8000
```

这表示只有服务器本机能访问 `127.0.0.1:8000`，公网不能直接访问 `服务器IP:8000`。当前正式部署应通过 Cloudflare Tunnel 访问域名。

如果没有域名，仅做临时 IP 验证，见 `docs/no-domain-access-notes.md`。

### 5. Cloudflare Tunnel 状态不是 HEALTHY

先检查服务器服务：

```bash
sudo systemctl status cloudflared --no-pager
sudo journalctl -u cloudflared -n 50 --no-pager
```

如果日志中能看到：

```text
API is reachable
Environment is healthy
```

但随后反复出现：

```text
control stream encountered a failure while serving
Retrying connection
```

先回到 Cloudflare 控制台查看 Tunnel 状态。如果控制台显示 `HEALTHY`，通常可以继续配置路由；如果显示 `Disconnected` 或 `Unhealthy`，再考虑切换协议或重装 connector。

### 6. 域名访问失败

按顺序检查：

```bash
curl --fail http://127.0.0.1:8000/readyz
sudo systemctl status cloudflared --no-pager
sudo journalctl -u cloudflared -n 50 --no-pager
curl -I https://你的域名
```

如果本机 `127.0.0.1:8000/readyz` 正常，但域名不正常，问题通常在 Cloudflare Tunnel、Published application route 或 Cloudflare DNS 路由，不在 Docker 容器。

### 7. Published application route 填错

根域名不要填在 `Subdomain` 里。错误示例：

```text
Subdomain: https://getwiseoracle.com
Domain: getwiseoracle.com
Full hostname: https://getwiseoracle.com.getwiseoracle.com
```

正确示例：

```text
Subdomain: 留空
Domain: getwiseoracle.com
Full hostname: getwiseoracle.com
Type: HTTP
URL: 127.0.0.1:8000
```

### 8. 传统 Nginx/Certbot 方案什么时候使用

如果不使用 Cloudflare Tunnel，而是让域名直接解析到服务器公网 IP，才需要在服务器开放 `80/443`、配置 Nginx、申请 Certbot 证书。当前源站 IP 已被 GFW ban，推荐继续使用 Cloudflare Tunnel。
